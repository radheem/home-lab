# Journal — Experiment 03 (2026-06-14, design)

This experiment is a **design + tooling** deliverable (no live run yet — needs an
API key). What was established:

## Findings carried from Exp 02
- `GET https://abc.yourcompany.com/api/v1/routes` with the node auth key → **401**.
- `/health` → 200 (control plane reachable from the cluster host).
- Our router node `qube-homelab-ts-router` (100.64.0.47) shows `PrimaryRoutes: None`
  (route advertised but not approved). Sibling `qube-genai-ts-router` shows
  `PrimaryRoutes: ["172.28.240.0/24"]` → proves the approved-route shape we want.

## Decisions
- Remote approval requires a **Headscale API key** (admin Bearer), provisioned once
  by someone with control-plane access. The node/pre-auth key cannot be promoted.
- Added `tailscale/approve-route.sh` — approves via API, auto-detecting the Headscale
  API generation (legacy `/api/v1/routes/{id}/enable` vs `/api/v1/node/{id}/approve_routes`).
- Documented the zero-touch alternative: `autoApprovers` ACL + a **tagged** auth key,
  so routes approve automatically at registration (best long-term; needs a one-time
  policy change).
