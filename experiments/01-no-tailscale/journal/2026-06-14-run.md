### Step 1 — install.sh
- **Attempt 1 FAILED** at cluster create:
  `docker failed to create new network 'k3d-qube-homelab': invalid pool request:
  Pool overlaps with other one on this address space`.
  - Cause: `docker_default` owns the entire `172.20.0.0/16`, and `.env` had
    `CLUSTER_SUBNET=172.20.50.0/24` (a subset) → overlap.
  - Also observed: this host is AWS EC2 (`ens5` @ `172.31.16.0/20`), so there is
    **no physical home LAN** here. L2-announced LB IPs on `172.28.210.0/24` won't
    be ARP-reachable from a real LAN; we'll verify the pipeline (a) in-cluster and
    (b) from the host via a static route to the k3d bridge.
  - Fix: `CLUSTER_SUBNET=172.21.50.0/24` (free). `LB_CIDR=172.28.210.0/24` is
    already free (no docker net / host route uses it).
- **Attempt 2**: re-ran `./install.sh`.

- **Attempt 2 SUCCEEDED**: `install.sh` exit 0. All phases green; both nodes Ready;
  wildcard cert Ready; Gateway `Programmed=True`.
- LB-IPAM pinning **worked exactly**: `coredns-auth` EXTERNAL-IP `172.28.210.53`,
  `cilium-gateway-shared-gateway` EXTERNAL-IP `172.28.210.80`.

### Step 2 — deploy echo
- `kubectl kustomize test/echo | envsubst '${LOCAL_TLD}' | kubectl apply -f -`
- HTTPRoute `Accepted=True ResolvedRefs=True`. Pod Running.

### Step 3 — verify DNS  (the hard part — three bugs found)
1. **ExternalDNS CrashLoop #1**: `failed to sync *v1.Namespace: context deadline
   exceeded`. Root cause: ClusterRole missing `namespaces` (the gateway-httproute
   source watches Namespaces). Fix: added `namespaces` to the core `""` rule.
2. **ExternalDNS CrashLoop #2**: `etcd-client ... context deadline exceeded` /
   `Failed to do run once`. Same symptom for CoreDNS-auth. Root cause isolated by
   matrix test (cross/same node × IP/hostname):
   | endpoint form | result |
   |---|---|
   | pod IP | ✅ healthy 2ms |
   | ClusterIP **by IP** | ✅ healthy (cross-node) |
   | any **hostname** (headless or ClusterIP), any node | ❌ DeadlineExceeded |
   The **etcd gRPC client mis-resolves k8s hostnames** in this env — the pod
   `resolv.conf` inherited the AWS host search domains (`us-east-2.compute.internal`,
   `yourcompany.lan`) + `ndots:5`. Fix: added a pinned-ClusterIP `etcd-client` (10.43.0.20)
   and pointed BOTH ExternalDNS (`ETCD_URLS`) and CoreDNS-auth (`endpoint`) at it
   **by IP**. Kept the headless `etcd` svc for StatefulSet governance.
- After fixes: etcd holds `/skydns/lan/home/echo -> 172.28.210.80` (+TXT owner).
  ExternalDNS log: `Add/set key /skydns/lan/home/echo ... Host=172.28.210.80`.
- `nslookup echo.home.lan @172.28.210.53` → `172.28.210.80` ✅
- `nslookup example.com @172.28.210.53` → real public IPs ✅ (upstream forward OK)
- Cluster-DNS forward also OK: `nslookup echo.home.lan @10.43.0.10` → 172.28.210.80.

### Step 4 — verify routing
- `curl --resolve echo.home.lan:80:172.28.210.80` → echo-server JSON with
  `host.hostname=echo.home.lan`, `x-envoy-internal:true` (Cilium Gateway). ✅
  Full path DNS→Gateway→Service→Pod confirmed.

### Step 5 — host-level dig/curl (LAN-client simulation)
- Operator added the route: `sudo ip route add 172.28.210.0/24 dev br-c8b91c8d0283`
  (k3d bridge, host = 172.21.50.1). Needed only on this AWS host (no bridged LAN NIC);
  a real home host ARPs the LB IPs on the LAN directly.
- After the route: host `curl` to the **Gateway** `172.28.210.80` worked immediately
  (echo JSON, `x-forwarded-for: 172.21.50.1`). But `dig @172.28.210.53` (DNS LB IP)
  first **timed out**, then after a config change **connection refused**.
- **Bug #5 — L2 cross-node LB forwarding.** Diagnosis:
  - ARP resolved both IPs (`.80`→server-0 MAC, `.53`→agent-0 MAC); both nodes pingable.
  - L2 leases: Gateway `.80`→**server-0** (works), coredns-auth `.53`→**agent-0**.
  - coredns-auth was a Deployment with both replicas on server-0, so the agent-0
    lease holder had to forward cross-node → that path is unreliable in k3d (timeout).
  - Fix: made coredns-auth a **DaemonSet** + Service **`externalTrafficPolicy: Local`**
    so the lease-holding node always has a local backend (and src IP is preserved).
  - The brief `connection refused` after switching was transient — the agent-0
    DaemonSet pod's LB frontend wasn't ready yet. Once ready, `dig` succeeded with
    the lease still on agent-0. Confirmed `cilium-dbg service list` shows the `.53`
    frontend with a local backend on **both** nodes.
- **Host verification (PASS):**
  - `dig @172.28.210.53 echo.home.lan` → `172.28.210.80`
  - `dig @172.28.210.53 whoami.home.lan` → `172.28.210.80`
  - `dig @172.28.210.53 example.com` → public IP (upstream forward)
  - `curl --resolve echo.home.lan:80:172.28.210.80 .../host-verify` → echo JSON, client=172.21.50.1
  - `curl whoami` via Gateway → whoami response
- Known quirk (non-blocking): in-cluster `curl` by NAME from the **musl/alpine**
  curl image returns 000 — musl `getaddrinfo` + inherited AWS search domains. Real
  LAN clients (glibc, no k8s search domains) resolve in one query (proven by `dig` above).

## Problems & solutions
| # | Problem | Root cause | Fix |
|---|---------|-----------|-----|
| 1 | `source .env` → `9.9.9.9: command not found` | unquoted space in `UPSTREAM_DNS` | quoted the value (`.env` + `.env.example`) |
| 2 | k3d create: network pool overlap | `docker_default` owns `172.20.0.0/16`; node subnet was a subset | `CLUSTER_SUBNET=172.21.50.0/24` |
| 3 | ExternalDNS CrashLoop (Namespace sync) | ClusterRole lacked `namespaces` | added `namespaces` to core rule |
| 4 | ExternalDNS + CoreDNS-auth can't reach etcd | etcd gRPC client mis-resolves k8s hostnames (inherited search domains + ndots) | pinned `etcd-client` ClusterIP, referenced **by IP** |
| 5 | DNS LB IP unreachable from host (timeout/refused) | L2 lease on a node with no local coredns-auth pod; cross-node LB forwarding unreliable in k3d | coredns-auth → **DaemonSet** + Service **`externalTrafficPolicy: Local`** |

All five fixes are committed to the repo manifests/.env, so a fresh `./install.sh`
reproduces the working state (the host route + `CLUSTER_SUBNET` are env-specific).

## Result
**PASS.** Zero-touch DNS verified end-to-end **from the host** (LAN-client
simulation): HTTPRoute hostname → ExternalDNS → etcd → authoritative CoreDNS
(`172.28.210.53`) → Gateway (`172.28.210.80`) → pod. `dig` + `curl` both succeed;
upstream forwarding works. 5 bugs found and fixed along the way.
